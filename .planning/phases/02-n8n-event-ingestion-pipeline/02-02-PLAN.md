<objective>
Build the `/api/internal/events` webhook endpoint that n8n calls when a Voice AI call starts and ends. The endpoint verifies an HMAC-SHA256 signature, resolves the tenant from the DID number (never from payload), inserts a `call_log` row on `call.started`, upserts the same row on `call.completed` (idempotent via `session_id`), and creates an `inbox_items` row when the call qualifies. Also ship the n8n Vault-key-bootstrap workflow export proving keys come from Vault as Node 1.

Purpose: This is the ingestion pipeline itself — the dashboard has no data without it (REALTIME-01, REALTIME-02). Success Criteria 1, 2, and 5 of Phase 2 are satisfied here.
Output: HMAC helper, Zod payload schemas, the POST route, and the n8n workflow JSON export.

This plan writes code only — it does NOT depend on the live DB (02-01 Task 3 push). It can be verified statically (tsc, build, unit-level HMAC test). Runtime DB behavior is verified in 02-03 / phase verification after the push.
</objective>

    Create `src/app/api/internal/events/route.ts` exporting `async function POST(request: NextRequest)`.

    Step 1 — Read raw body once: `const rawBody = await request.text()`. The HMAC must be verified against the exact raw bytes, so do NOT call `request.json()` first.

    Step 2 — Load the webhook secret: `const supabase = createServiceRoleClient()`, then `const { data: secret } = await supabase.rpc('get_secret', { secret_name: 'n8n_webhook_secret' })`. If no secret is returned, respond `500 { ok:false, error: 'Webhook-Konfigurationsfehler.' }`.

    Step 4 — Parse + validate: `JSON.parse(rawBody)` inside try/catch (catch → `400 { ok:false, error:'Ungültiger JSON-Body.' }`), then `webhookEventSchema.safeParse(...)` (failure → `400 { ok:false, error:'Ungültige Ereignisdaten.' }`).

    Step 6 — Upsert call_log. For BOTH event types use `supabase.from('call_log').upsert({...}, { onConflict: 'session_id' })` — never a plain INSERT (idempotency, C10).

Chain `.select('id').single()` to get the call_log id back.
    - On a DB error → `500 { ok:false, error:'Speicherung fehlgeschlagen.' }`.

'technical_error', status: 'open' }, { onConflict: 'call_log_id' })`. The `inbox_items.call_log_id` column carries a `UNIQUE` constraint (added in the 02-01 migration), so this `onConflict: 'call_log_id'` upsert is idempotent — re-sending the same call.completed event leaves exactly one inbox item per call.

    Step 8 — Respond `200 { ok:true, call_log_id: callLogId }`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| n8n → /api/internal/events | untrusted HTTP crosses here; only an HMAC signature proves origin |
| webhook payload → DB | payload fields are attacker-controllable; tenant_id must NOT be trusted from payload |
| n8n node config → Supabase Vault | secrets must live in Vault, fetched at runtime — never inline in node JSON |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Spoofing | forged webhook POST to /api/internal/events | mitigate | HMAC-SHA256 verification of raw body against Vault `n8n_webhook_secret`; missing/invalid → 401 before any DB access |
| T-02-06 | Tampering | payload supplies a foreign tenant_id (cross-tenant poisoning, C9) | mitigate | Schema has no tenant_id field; tenant resolved exclusively via `tenant_did_numbers` SELECT by did_number; unknown DID → 404 |
| T-02-07 | Tampering | duplicate ElevenLabs events create duplicate call_log rows (C10) | mitigate | `upsert(..., { onConflict: 'session_id' })` with UNIQUE(session_id); replay produces exactly one row |
| T-02-08 | Replay | timing attack on HMAC comparison | mitigate | `crypto.timingSafeEqual` for signature comparison |
| T-02-09 | Information Disclosure | transcript_text / phone_hash leaked into logs | mitigate | No logging of transcript_text or phone_hash anywhere in the route |
| T-02-SC | Tampering | n8n workflow stores raw service-role / provider keys in node config | mitigate | n8n Node 1 fetches all keys via `get_secret` from Vault; verify step greps the export for literal secrets and fails the build if found |
</threat_model>
