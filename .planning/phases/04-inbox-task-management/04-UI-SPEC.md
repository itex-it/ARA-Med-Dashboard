| open | in_progress | "Fall annehmen" | "Annehmen" | default |

| State | Visual |
|-------|--------|

| Transition confirmed (Realtime) | Status chip settles to confirmed value |
| Transition error | Status chip reverts, toast: "StatusГ¤nderung fehlgeschlagen. Bitte erneut versuchen." |

### Note Save (INBOX-05)

| State | Visual |
|-------|--------|
| Idle | Textarea editable, "Notiz speichern" button active |
| Saving | Button disabled, spinner, textarea disabled |
| Saved | Button re-enables, no toast (silent success вЂ” note content is visual confirmation) |
| Error | Toast: "Notiz konnte nicht gespeichert werden." |

### Real-time Case Update

| Event | Visual |
|-------|--------|
| New case arrives (INSERT) | Row slides in from top (`animate-in slide-in-from-top-2`) in currently active tab |
| Status changes (UPDATE) | Row status chip transitions in-place (no layout shift); tab count badges update |
| Note added by another user (UPDATE) | If sheet is open for that case: textarea content refreshes silently |

## Implementation Notes for Executor

1. **Route:** Create `src/app/(dashboard)/inbox/page.tsx` as a Server Component. Reads `tenantId` from JWT, passes `tenantId` + permission flags to `<InboxTable>` Client Component.

2. **Tab filter via URL params:** Use `searchParams` to read `filter` param in the Server Component. Pass initial `filter` to `<InboxTable>`. Tab changes update URL without full navigation (`router.push` with shallow).

3. **InboxTable Client Component:** `src/components/inbox/InboxTable.tsx`. Uses `useRealtimeChannel` on table `inbox_items` (same hook pattern as `CallLogTable`). Initial load: Server Action or direct Supabase query tenant-scoped, ordered by `created_at DESC`. Filter applied client-side on the `status` field (tab switch does not re-fetch вЂ” filter from in-memory rows).

4. **CaseDetailSheet Client Component:** `src/components/inbox/CaseDetailSheet.tsx`. Accepts `InboxItemRow` + `hasEditRight: boolean` + `hasManageRight: boolean` + `hasCallDetail: boolean` as props. Never fetches permissions вЂ” receives them as typed props from Server Component.

5. **Status transition Server Action:** `src/app/actions/update-inbox-status.ts` вЂ” updates `inbox_items.status`. Validates tenant_id from JWT. Returns typed state. Client uses `useTransition` + optimistic update.

6. **Note save Server Action:** `src/app/actions/save-inbox-note.ts` вЂ” updates `inbox_items.internal_note`. Same pattern. No toast on success (textarea content is the visual confirmation).

7. **OpenTaskCounter link:** Wrap the count badge in the StatusBar in `<Link href="/inbox?filter=open">`. This requires no architecture change вЂ” StatusBar is already a Client Component.

8. **Emergency row highlight:** Case rows with `case_type === 'emergency'` get `border-l-2 border-destructive` regardless of selection state. Selected non-emergency rows get `border-l-2 border-primary`.

9. **Tab count badges:** Maintained in Client Component local state, initialized from query count, updated via Realtime deltas (same pattern as `useOpenTaskCount` in Phase 3).

10. **No pagination for MVP:** Show last 200 inbox items ordered by `created_at DESC`. Add "Г„ltere laden" button only if > 200 rows exist.
