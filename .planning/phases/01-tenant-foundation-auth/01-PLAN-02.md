    Create supabase/migrations/20260522000003_rls_policies.sql:

    Four separate RLS policies for user_tenant_roles (per D-17 — one per operation):

    SELECT policy "utr_tenant_isolation_select": FOR SELECT TO authenticated USING ((SELECT auth.jwt()->'app_metadata'->>'tenant_id')::uuid = tenant_id). This lets users see only their tenant's rows.

    INSERT policy "utr_tenant_isolation_insert": FOR INSERT TO authenticated WITH CHECK ((SELECT auth.jwt()->'app_metadata'->>'tenant_id')::uuid = tenant_id). Prevents a user inserting a row for another tenant.

    UPDATE policy "utr_tenant_isolation_update": FOR UPDATE TO authenticated USING ((SELECT auth.jwt()->'app_metadata'->>'tenant_id')::uuid = tenant_id AND (SELECT auth.jwt()->'app_metadata'->>'ara_role') IN ('operator', 'ordination_admin')). Only operators and admins can update role assignments.

    DELETE policy "utr_tenant_isolation_delete": FOR DELETE TO authenticated USING ((SELECT auth.jwt()->'app_metadata'->>'tenant_id')::uuid = tenant_id AND (SELECT auth.jwt()->'app_metadata'->>'ara_role') IN ('operator', 'ordination_admin')). Only operators and admins can delete role assignments.

    IMPORTANT: Never use the bare `auth.jwt()->'app_metadata'->>'tenant_id'` without the `(SELECT ...)` wrapper — the subquery form is what causes Postgres to cache the result once per statement.

    ---

    Create supabase/migrations/20260522000004_access_token_hook.sql:

    This is the Custom Access Token Hook. It runs at every token issuance (login, refresh) and injects tenant_id and ara_role into app_metadata.

    Function signature: `CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb) RETURNS jsonb LANGUAGE plpgsql STABLE SECURITY DEFINER`

    Function body (per D-01 and D-02):
    1. Declare v_tenant_id uuid, v_role text, claims jsonb
    2. SELECT tenant_id, role INTO v_tenant_id, v_role FROM public.user_tenant_roles WHERE user_id = (event->>'user_id')::uuid AND active = true LIMIT 1
    3. claims := event->'claims'
    4. IF v_tenant_id IS NOT NULL THEN: SET app_metadata.tenant_id and app_metadata.ara_role using jsonb_set
    5. event := jsonb_set(event, '{claims}', claims); RETURN event;

    Grants: GRANT EXECUTE ON FUNCTION public.custom_access_token_hook TO supabase_auth_admin; GRANT USAGE ON SCHEMA public TO supabase_auth_admin;

    Security: REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM PUBLIC, authenticated, anon;

    Note in a comment: After applying this migration, the Supabase Auth hook must be registered in the Supabase Dashboard under Authentication > Hooks > Custom Access Token Hook. Point it to `public.custom_access_token_hook`.

    ---

    Create supabase/migrations/20260522000005_vault_helper.sql:

    Create function public.get_secret(secret_name text) RETURNS text LANGUAGE sql SECURITY DEFINER as:
    SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = secret_name LIMIT 1;

    Grants and revokes:
    REVOKE ALL ON FUNCTION public.get_secret(text) FROM PUBLIC, authenticated, anon;
    GRANT EXECUTE ON FUNCTION public.get_secret(text) TO service_role;

    This function is the ONLY way to read Vault secrets. n8n calls it via its Postgres service-role connection.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Auth hook → app_metadata | Server-writable only; user cannot modify app_metadata |
| RLS subquery cache | Postgres caches (SELECT auth.jwt()...) per statement, not per row |
| Vault helper | Only service_role can call get_secret(); authenticated and anon are explicitly revoked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | RLS policy reads tenant_id from request body | mitigate | RLS policies read exclusively from JWT app_metadata via the SELECT subquery; no request body involved |
| T-02-02 | Tampering | Bare `auth.jwt()` call causes per-row evaluation | mitigate | All RLS policies use `(SELECT auth.jwt()...)` wrapper — enforced by acceptance criteria grep check |
| T-02-03 | Information Disclosure | vault.decrypted_secrets accessible to authenticated role | mitigate | get_secret() REVOKES from authenticated/anon; vault table itself not directly accessible |
| T-02-04 | Elevation of Privilege | Custom hook reads user_metadata (user-modifiable) | mitigate | Hook reads app_metadata (server-writable only) from user_tenant_roles table, not user_metadata |
| T-02-05 | Elevation of Privilege | Role not validated in CHECK constraint | mitigate | role column has CHECK (role IN ('operator', 'ordination_admin', 'assistant', 'viewer')) |
| T-02-SC | Tampering | Migration SQL tampering | accept | SQL files are version-controlled in git; Supabase applies them in order; diffs are reviewable |
</threat_model>
