<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /auth/login | Credentials submitted over HTTPS; server validates with Supabase Auth |
| Server Action → Supabase Auth | Uses server client with anon key + cookie; no service-role key used for auth operations |
| /api/admin/revoke-session | System-to-system endpoint; protected by service-role key in Authorization header |
| TOTP Factor → JWT aal claim | Supabase sets aal='aal2' in session after successful verifyMFA; proxy.ts reads it |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | Login form submits to wrong endpoint | mitigate | Server Action is bound to the login page form; no direct POST endpoint exposed |
| T-03-02 | Information Disclosure | Error message reveals whether email exists | mitigate | loginAction returns identical error message regardless of failure reason: "Ungültige E-Mail-Adresse oder Passwort." |
| T-03-03 | Elevation of Privilege | Operator bypasses TOTP by direct URL access | mitigate | proxy.ts enforces AAL2 redirect for operator/ordination_admin on every request — URL bypass is impossible |
| T-03-04 | Tampering | Password reset link reused after use | accept | Supabase Auth tokens are single-use by design; Supabase enforces this |
| T-03-05 | Elevation of Privilege | revoke-session called by unauthorized actor | mitigate | POST /api/admin/revoke-session requires Bearer token matching SUPABASE_SERVICE_ROLE_KEY |
| T-03-06 | Information Disclosure | getSession() used in Server Component (cookie-spoofable) | mitigate | All server code uses getUser() which re-validates with the Auth server; getSession() is never imported |
| T-03-SC | Tampering | npm/shadcn package install | mitigate | shadcn add installs official shadcn components only; all packages are from established publishers |
</threat_model>
