    Create supabase/migrations/20260523000020_phase5_seed_appointment_types.sql.

    The seed must insert rows without a concrete tenant_id (the appointment_types table requires tenant_id NOT NULL). The approach: create a temporary SECURITY DEFINER function that inserts the rows for every existing tenant in the tenants table, then drop the function. This runs once at migration time and covers any tenants already in the system.

    Structure:

    DO $$ 
    BEGIN
      -- Insert 7 common Austrian GP appointment types for all existing tenants.
      -- Uses INSERT ... ON CONFLICT DO NOTHING so re-running is safe.
      INSERT INTO public.appointment_types 
        (tenant_id, appointment_type_code, display_name, is_visible, is_voice_bookable, is_internal_only, is_default, pid_zero_allowed)
      SELECT 
        t.id,
        v.appointment_type_code,
        v.display_name,
        true,   -- is_visible
        false,  -- is_voice_bookable (operator configures per-practice)
        false,  -- is_internal_only
        v.is_default,
        v.pid_zero_allowed
      FROM public.tenants t
      CROSS JOIN (VALUES
        ('ALLG',   'Allgemeinuntersuchung',  false, false),
        ('BLUT',   'Blutabnahme',            false, false),
        ('KIND',   'Kinderuntersuchung',     false, false),
        ('HAUS',   'Hausbesuch',             false, false),
        ('REZEPT', 'Rezeptausstellung',      false, false),
        ('UEBERW', 'Überweisung',            false, false),
        ('IMPF',   'Impfung',               false, false)
      ) AS v(appointment_type_code, display_name, is_default, pid_zero_allowed)
      ON CONFLICT (tenant_id, appointment_type_code) DO NOTHING;
    END $$;

    Important notes:
    - The CROSS JOIN ensures every existing tenant gets all 7 rows — if no tenants exist yet, no rows are inserted (that is fine; tenant creation flow must seed separately, or run a future seeding step post tenant-creation).
    - ON CONFLICT DO NOTHING makes this idempotent — safe to re-apply.
    - is_voice_bookable=false on all rows: the operator must explicitly enable AI-booking per type; this is the secure default.
    - is_default=false on all rows: operator sets the default for unclear complaints via the Terminarten UI (APPT-04). Allgemeinuntersuchung is the likely choice but must be a deliberate operator decision.
    - pid_zero_allowed=false on all rows by default. The operator enables it for types like Führerscheinuntersuchung (APPT-05) if applicable to their practice.
    - No hardcoded tenant UUID in this migration — the DO block reads from public.tenants dynamically.
    - This migration runs after 20260523000018 (tables) and 20260523000019 (RLS) in migration order.
  </action>

| Boundary | Description |
|----------|-------------|
| Browser → Server Action | Configuration form inputs are untrusted; validated by Zod before use |
| Server Action → Supabase | Service-role client bypasses RLS; tenant_id injected from JWT only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | All Phase 5 Server Actions | mitigate | tenant_id always from user.app_metadata.tenant_id — never from formData or request body (per STATE.md C2) |
| T-05-02 | Tampering | service-role DELETE/UPDATE | mitigate | All deletes include .eq('tenant_id', tenantId) — service-role bypasses RLS so explicit filter is mandatory defense-in-depth |
| T-05-03 | Tampering | EU AI Act disclosure (greeting_texts) | mitigate | Server Action unconditionally overwrites eu_ai_act_disclosure column with locked constant; eu_ai_act_disclosure_present always set true — UI lock is UX only |
| T-05-04 | Tampering | appointment_types mutual exclusivity (is_voice_bookable vs is_internal_only) | mitigate | Zod .refine() in Server Action: cannot have both true simultaneously |
| T-05-05 | Tampering | Date range reversal (end_date < start_date) | mitigate | Zod .refine() in Server Actions + DB CHECK constraint on deputy_periods and deputy_doctors |
| T-05-06 | Tampering | PZN format injection (medications) | mitigate | Zod z.string().regex(/^\d{7}$/) in Server Action |
| T-05-07 | Elevation | Config mutations without role gate | mitigate | All write Server Actions check ara_role IN ('operator','ordination_admin') before any DB write |
| T-05-08 | Information Disclosure | RLS per-row JWT call (C1 pitfall) | mitigate | All 36 RLS policies use (SELECT auth.jwt()->'app_metadata'->>'tenant_id')::uuid — cached subquery form, never bare auth.jwt() per-row |
| T-05-SC | Tampering | shadcn add commands (registry fetch) | accept | shadcn official registry only; no third-party registries; generates local source files, not npm packages — no supply chain risk beyond existing shadcn dependency |
