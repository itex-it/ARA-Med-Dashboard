## Spacing Scale

Identical to Phase 5 — 8-point scale, multiples of 4:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge padding, channel icon spacing |
| sm | 8px | Compact field spacing, button gaps within a row, tag/chip gaps |
| md | 16px | Default vertical spacing between fields; card inner padding |
| lg | 24px | Section padding (p-6 matches established layout pattern) |
| xl | 32px | Gap between configuration sub-sections |
| 2xl | 48px | Gap between major module areas (Routing vs. Kommunikation) |
| 3xl | 64px | Page-level vertical padding (not used — layout provides p-6) |

Exceptions:
- Touch targets for action radio buttons in routing rule form: minimum 44px height
- Send log table rows: minimum 40px height for readability of masked recipients

## Page Structure

### Decision: Extend /konfiguration with two new tabs

Phase 6 adds two tabs to the existing `KonfigurationTabs` at `/konfiguration`:
- "Routing" (tab value: `routing`)
- "Kommunikation" (tab value: `kommunikation`)

**Rationale:** All configuration lives under `/konfiguration`. Adding a second page (`/routing-kommunikation`) would split the configuration mental model. Phase 5 already established the `KonfigurationTabs` extension pattern. Phase 6 follows it exactly.

The existing `KonfigurationTabs` component is extended. The `konfiguration/page.tsx` Server Component fetches Phase 6 data alongside existing Phase 5 data in the same `Promise.all`.

```
/konfiguration
  └── Tabs (horizontal, top-level navigation):
        ├── Öffnungszeiten      (Phase 5 — unchanged)
        ├── Terminarten         (Phase 5 — unchanged)
        ├── Begrüßungstexte     (Phase 5 — unchanged)
        ├── Vertretung          (Phase 5 — unchanged)
        ├── Medikamente         (Phase 5 — unchanged)
        ├── Routing             (Phase 6 — NEW: ROUTE-01..03)
        └── Kommunikation       (Phase 6 — NEW: COMM-01..05)
```

Tab values: `routing` | `kommunikation`

URL sync: same `?tab=` search param pattern as Phase 5.

**File locations:**
- New tab components: `src/components/config/RoutingTab.tsx` + `src/components/config/KommunikationTab.tsx`
- Server Actions: `src/lib/actions/routing.ts` + `src/lib/actions/communication.ts`
- Types: extended `src/lib/types.ts`

#### Sub-tab 4 — Versandprotokoll (COMM-04)

**Purpose:** Read-only log of every send attempt — visible to operator/ordination_admin.

**DSGVO constraint:** Recipient phone numbers and email addresses are always shown masked. Never show full contact details.

**Masking format:**
- Phone: "+43 *** *** 789" (last 3 digits visible)
- Email: "m***a@beispiel.at" (first char + domain visible)

**List view:**

| Element | Spec |
|---------|------|
| Layout | `Table` with columns: Zeitpunkt, Ereignis, Kanal, Empfänger (maskiert), Vorlage, Status, Fehler |
| Zeitpunkt | German datetime: dd.MM.yyyy HH:mm — via `Intl.DateTimeFormat('de-AT')` |
| Ereignis | text-sm — event type German label |
| Kanal | Channel icon + label (same as rules list) |
| Empfänger | `font-mono text-sm text-muted-foreground` — masked value |
| Vorlage | text-sm text-muted-foreground — template name + version (e.g. "Terminbestätigung DE v2") |
| Status | `Badge` with semantic color (Zugestellt = green, Ausstehend = yellow, Fehlgeschlagen = red) |
| Fehler | text-xs text-muted-foreground — error reason if status = Fehlgeschlagen; "—" otherwise |

**Filtering:**

Above the table, a filter bar:
- `Select` "Alle Ereignisse" — filters by event type (German labels)
- `Select` "Alle Kanäle" — filters by channel
- `Select` "Alle Status" — options: "Alle", "Zugestellt", "Ausstehend", "Fehlgeschlagen"
- Date range: `Popover` with `Calendar` for Von date + `Popover` with `Calendar` for Bis date
- Filter bar layout: horizontal flex row, gap-2, filters right-aligned

**Pagination:** 25 rows per page. `Button` variant="outline" size="sm" "Ältere laden" at bottom — cursor-based pagination.

**Empty state:** "Noch keine Versandversuche protokolliert."

**No edit controls on this sub-tab — read-only.**

## Component Patterns

### Form Save Pattern (unchanged from Phase 5)

- Page: Server Component — reads JWT, passes `tenantId` + `hasEditRight` as props
- Tab component: Client Component (`'use client'`) — renders Sheet + Table
- Form inside Sheet: Client Component (`'use client'`) — `useActionState` (React 19)
- Server Action: validates `tenantId` from JWT (never from form body), validates `araRole` from JWT against `['operator', 'ordination_admin']`, runs Zod schema, upserts/inserts to Supabase
- Pending: `Loader2 className="h-3 w-3 animate-spin mr-1"` inside Button
- Error: `<p className="text-xs text-destructive mt-1">{state.error}</p>` below CTA

### Section Header Pattern (locked — matches Phases 3/4/5)

```tsx
<p className="text-xs font-semibold uppercase tracking-wide text-muted-foreground mb-2">
  {sectionTitle}
</p>
```

### Sheet Pattern (for routing rules and communication rules)

- Use `Sheet` (already installed) — not Dialog — for rule add/edit (Sheet is for full detail views; Dialog is for confirmations only)
- Sheet trigger: `Pencil` icon button on existing row OR "hinzufügen" button
- Sheet closes after successful save (state.ok = true triggers `setOpen(false)`)
- Sheet does not close on outside click when form is dirty (prevent accidental data loss) — use `SheetContent onInteractOutside={(e) => { if (isDirty) e.preventDefault() }}`

### Switch Toggle Pattern (matches Phase 3/5 — rule active/inactive)

- `Switch` with `useOptimistic` + `useTransition`
- Spinner `Loader2` shown during pending
- Switch disabled during transition

### Inline Add Form Pattern (VIP numbers)

Same pattern as Phase 5 inline adds (medications, special days): form row at bottom of table, not a Sheet.

### Nested Tabs Pattern (Kommunikation sub-tabs)

```tsx
<Tabs defaultValue="intern" className="w-full">
  <TabsList className="mb-4">
    <TabsTrigger value="intern">Interne Benachrichtigungen</TabsTrigger>
    <TabsTrigger value="patient">Patientenbenachrichtigungen</TabsTrigger>
    <TabsTrigger value="vorlagen">Nachrichtenvorlagen</TabsTrigger>
    <TabsTrigger value="protokoll">Versandprotokoll</TabsTrigger>
  </TabsList>
  ...
</Tabs>
```

Sub-tab state is NOT synced to URL (only top-level tab `?tab=kommunikation` synced). Sub-tab state is local React state.

### Read-Only Send Log Pattern

- No edit controls on Versandprotokoll sub-tab
- `hasEditRight` prop is still passed to component (needed for role gate check on initial data fetch), but no add/edit/delete UI rendered
- Filter controls visible to all roles (viewer can filter the log they can see)

## Accessibility Constraints

- All `Switch` components: `aria-label` with descriptive German label (e.g. "Routing-Regel aktiv: VIP-Anrufer")
- Sheet: focus trap via Radix (shadcn Sheet provides this)
- Confirmation dialogs: focus trap inside Dialog (shadcn Dialog provides this via Radix)
- Error messages: `role="alert"` on error paragraphs
- Variable insert buttons in template editor: `aria-label="Variable {{patient_name}} einfügen"`
- Send log table: `aria-label="Versandprotokoll"` on `<Table>`
- Masked recipient cells: `aria-label="Maskierte Empfängeradresse"` — screen readers must not attempt to read the masking asterisks as content
- Channel icon badges: icon has `aria-hidden="true"`, text label is visible — no icon-only badges

---

## Security Constraints (UI-Level)

- All CRUD Server Actions for routing rules and communication rules check `araRole` from JWT against `['operator', 'ordination_admin']` — return 403 if not met
- `hasEditRight` prop controls whether add/edit/delete controls render — role check at server action is the security gate; UI is defense-in-depth only
- Send log recipient column always masked at DB query level — never unmasked in the UI, no permission toggle for unmasking (DSGVO)
- Template preview substitution uses only hardcoded example values — never fetches real patient data for preview
