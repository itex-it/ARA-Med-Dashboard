## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Server Action | All routing and comm rule inputs are untrusted; validated by Zod before use |
| Server Action → Supabase | Service-role client bypasses RLS; tenant_id injected from JWT only |
| Migration → Postgres | SQL runs as migration user; no dynamic input — safe |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing | All Phase 6 Server Actions (Plans 02-04) | mitigate | tenant_id always from user.app_metadata.tenant_id — never from formData or request body (per STATE.md C2 pitfall) |
| T-06-02 | Tampering | service-role DELETE/UPDATE on routing_rules, comm_rules, message_templates | mitigate | All deletes include .eq('tenant_id', tenantId) — service-role bypasses RLS, explicit filter is mandatory defense-in-depth |
| T-06-03 | Information Disclosure | send_log recipient column | mitigate | No raw recipient column in schema — only recipient_masked (TEXT). Masking applied at INSERT time in Server Action (Plan 06-04) using "+43 *** *** 789" format for phone and "m***a@beispiel.at" format for email |
| T-06-04 | Elevation | Config mutations without role gate | mitigate | All write Server Actions (Plans 02-04) check ara_role IN ('operator','ordination_admin') before any DB write |
| T-06-05 | Information Disclosure | RLS per-row JWT call (C1 pitfall) | mitigate | All 20 RLS policies use (SELECT auth.jwt()->'app_metadata'->>'tenant_id')::uuid — cached subquery form, never bare auth.jwt() per-row |
| T-06-06 | Tampering | JSONB condition_value / action_value injection in routing_rules | mitigate | Zod validates structure of condition_value and action_value in Server Action before upsert — DB accepts any JSONB but action layer enforces schema |
| T-06-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan — migration files and TypeScript only |
</threat_model>
